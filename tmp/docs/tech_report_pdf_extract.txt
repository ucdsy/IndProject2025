面向互联网基础资源的高可信语义路由
结构化判别与职责化协作复核
匿名稿
摘要
本文研究固定能力命名空间下的高可信语义路由任务：给定自然语言请求，系统需要在候选集合内部选出主
能力、恢复相关能力，并在必要时解析到具体执行实例。该任务不同于开放式文本生成，其难点在于跨域语
义重叠、层级粒度竞争、多意图表达和治理约束同时存在，而最终决策还必须具备可复核证据和可审计过程。
现有单模型直接裁决往往能够给出答案，但在边界样本上稳定性不足，也难以说明误判或改判的具体依据。
为此，本文提出一套受约束的协作式路由框架，由候选召回、规则路由、结构化语义判别、职责化协作
复核和执行落点解析组成。直接路由阶段在冻结候选集内生成候选级结构化判断，并与规则分数融合；协作
复核阶段则通过角色专属证据视图、共识排序和显式改判授权处理低置信与高风险样本。整个推断过程以统
一 trace contract 落盘，以支持样本级回放、责任归因和审计分析。
本文构建了覆盖 9 类领域、25 个能力基座的 563 条样本数据资产， 并在retrospective 协议下冻结为统一
train/test 主评测。实验显示，规则路由在 test 集上的准确率为 0.7876，结构化语义判别提升至 0.8761，
加入协作复核后进一步达到 0.8850，test acceptable accuracy 为 0.9115。若在训练划分上选择更积极的
升级策略并回放到同一冻结 test 集，系统可进一步达到 0.9292，并获得 5 / 5 / 0 的 Changed / Fixed /
Regressed 诊断结果；在 holdout3 对齐子集上，同样的职责化强放行将结果从 0.8625 提升到 0.9250。
这些结果表明，高可信语义路由的关键不在于引入开放式多代理讨论，而在于在固定候选边界内联合建
模结构化语义证据、职责化复核和改判授权。
关键词：语义路由；智能体命名；多智能体协作；可审计决策；互联网基础资源
1 引言
随着大模型系统开始处理可调用能力、服务实例和受约束任务入口，互联网基础资源场景中的
核心问题已经从“生成一段合理文本”转向“把请求稳定映射到正确的能力地址” 。 在这一设定下， 用户
输入往往同时携带主任务、次要诉求、场景线索和风险约束，系统不仅要在固定命名空间中做出主
能力判断，还要恢复相关能力并落到可执行实例。这个问题看似接近意图分类，实质上更接近受约
束决策：输出空间固定，候选之间存在层级和语义耦合，错误必须能够定位到召回、候选内判断或
执行落点的具体阶段。
现有做法通常落在两个极端。一类方法依赖规则或轻量打分器，在高精度关键词命中条件下能
够快速给出结果，但面对跨域重叠、父子节点冲突和多意图样本时容易失去区分能力。另一类方法
直接让单个大模型一步产出最终路由，虽然具备更强的语义泛化能力，却往往缺乏稳定的候选内比
较机制，也难以解释为什么改判某个 incumbent、为什么保留另一个竞争候选，或者为什么某次复
核被触发而另一次没有。对于互联网基础资源系统，这类不可复核的决策链会直接限制模型结果在
工程侧的可信使用。
本文将这一问题定义为高可信语义路由：系统首先在固定能力命名空间内召回候选，再在候选
集合内部完成主能力与相关能力判断，并在必要时触发职责化协作复核，最后将能力地址解析到具
体执行实例。围绕这一任务，我们提出一套受约束的协作式路由框架，将语义判别、升级触发和改
判授权拆成可独立分析的决策环节：直接路由阶段产生候选级结构化判断和不确定性摘要，协作复

2 相关工作与问题定位
与本文最接近的一类工作是 LLM routing 。RouteLLM、MixLLM 等方法研究的是按查询在多
个模型之间进行分配， 并围绕质量、 成本和延迟学习路由策略[6, 7]。 这类工作关心的是模型级选择，
而本文处理的是能力级语义路由： 系统必须在单一能力命名空间内选出正确的routing_fqdn， 同时
恢复 related 能力并维持改判边界的可审计性。因此，两者虽然都属于 routing，但决策对象、错误
来源和评价目标并不相同。 模型路由优化的是质量–成本权衡； 本文优化的是固定候选边界内的主能
力判断、related 恢复和协作改判质量。
第二类相关工作来自大模型推理增强。Chain-of-Thought、Self-Consistency 和 Tree of Thoughts
证明了多路径推理、 采样一致性和显式搜索能够提升复杂推理表现[1, 2, 3]。 这些方法的重要启发在
于：当任务存在多个竞争解释时，单一步骤推断往往不足，需要把中间判断显式展开。然而，上述
工作大多面向开放输出空间，较少要求系统在冻结候选集内完成主标签、相关标签和风险边界的联
合决策。本文借鉴了显式保留中间语义判断的思想，并将其进一步落实到固定候选边界内的受约束
决策。
第三类相关工作来自多智能体讨论与反馈式协作 [4]. 这一路线的核心假设是，多视角复核可以
缓解单模型的局部盲点。与此同时，近期研究也表明，多智能体讨论的收益并不稳定，控制协议和

3 方法
3.1 问题表述
本文区分两类核心语义对象。 第一类是能力地址routing_fqdn， 用于表示请求应被哪一类能力
处理； 第二类是实例地址agent_fqdn， 用于表示在该能力下由哪个具体实例执行。 能力地址要求语
义稳定，实例地址允许随运行环境和服务状态动态变化。通过显式区分这两个层级，系统可以把“语
义路由是否正确”和“执行落点是否合适”分开建模。
设能力命名空间为 N = {c1, . . . , cM }， 输入请求表示为x = (q, m)， 其中q 为自然语言查询，m
为可选上下文元信息，如领域、场景和风险线索。系统输出记为

实例。主评测集中在 ˆy(x) 与 ˆR(x) 的正确性，执行落点解析作为下游步骤独立建模。
方法设计遵循一个关键约束：所有下游决策都必须发生在候选召回返回的集合内部。记候选召

错误；若真值已被召回但最终选择错误，则属于候选内裁决错误。本文所有模块设计和实验分析都
建立在这一区分之上。
在推断层面，系统先由召回模块构造候选证据包，再由直接路由路径产生初始主能力与相关能
力预测，随后由升级策略决定是否触发协作复核，最后在必要时将能力地址解析到实例地址。形式
上，本文将这一流程写为
CR(x) → zA(x) → g(x) → zB(x) → ˆa(x), (3)
其中 zA(x) 表示直接路由输出，g(x) ∈ { 0, 1} 表示是否升级到协作复核，zB(x) 表示协作复核后的
最终路由结果。这样，系统增量可以被进一步归结为三个可分解问题：候选集合是否覆盖真值、主
能力是否在候选内被正确选出，以及相关能力与执行落点是否被稳定恢复。

3.2 候选召回
候选召回模块的目标是在能力命名空间中组织高覆盖、 低泄漏的候选集合。 本文采用descriptor-
only 的 clean recall 方案，仅使用节点描述、别名、层级信息与元数据标签完成词法和元数据融合
召回，以避免把下游决策信号提前泄漏到召回阶段。
对每个候选 c ∈ N ，召回模块计算
SR(c | x) = λ1ϕalias(q, c) + λ2ϕdesc(q, c) + λ3ϕhier(q, c) + λ4ϕmeta(m, c) − λ5ϕover(c), (4)
其中 ϕalias 表示查询与别名的匹配强度， ϕdesc 表示查询与节点描述的匹配强度， ϕhier 表示层级一
致性信号，ϕmeta 表示场景元信息与节点标签的匹配，ϕover 表示过度具体或层级失衡带来的惩罚项。
系统按 SR(c | x) 排序并截取 top-K 作为候选集合 CR(x)。
召回模块的输出是一份可供后续模块复用的候选证据包
ER(x) = {(c, sR(c), dc, ac, hc, mc) | c ∈ CR(x)}, (5)
其中 dc、ac、hc 和 mc 分别表示描述、别名、层级位置与元数据快照。由于后续所有模块都只能访
问 ER(x) 中的候选，召回模块实际定义了整个系统的搜索边界。定量召回统计放在实验部分报告；
从本文证据链来看，后续主要增量来自候选内判断质量，而非候选覆盖不足。

主能力候选、related 候选、 候选级task fit / primary fit / related fit 、 粒度判断、 风险失配标记， 以
及用于下游协作复核的语义交接信息。其输出写为

3.5 职责化协作复核
职责化协作复核 （Collaborative Review ） 负责处理低置信、margin 过小、 高风险和多意图冲突
样本。 它在候选集合内部执行职责约束下的复核， 而不重新打开开放式输出空间。 本文采用四类职责
角色：DomainExpert 负责任务匹配，GovernanceRisk 负责治理与风险放行，HierarchyResolver
负责层级与粒度冲突，UserPreference 负责主次意图拆分与偏好线索。

结果，因此本文的主结论当前聚焦于高可信语义路由本身。
这些限制也给出了后续最直接的扩展方向： 补充独立的前瞻性验证集， 统一质量–成本日志， 并
将当前冻结 lineage 整理为更完整的补充材料， 以便每一项结论都能从原始trace 精确回溯到最终表
格与图示。
8 结论
本文围绕互联网基础资源场景中的高可信语义路由问题， 提出了一套由候选召回、 规则路由、 结
构化语义判别、职责化协作复核和执行落点解析组成的受约束协作框架。围绕 563 条样本的数据资
产和冻结 train/test 主协议，实验表明：结构化语义判别建立了主要性能台阶，职责化协作复核
则在困难样本和更积极的升级策略下进一步释放净修正收益。
本文的核心结论是，高可信语义路由的提升不能简单归因于“增加更多代理”或“增加更多复核” ，
而必须依赖固定候选边界内的结构化语义证据、职责化角色分工以及可审计的改判授权机制共同作
用。在这一前提下，系统既能提高准确率，也能为每一次改判保留可回放、可归因的证据链。
A 附录：Prompt Contract 与结构化字段
A.1 直接路由的 Decision Packet
结构化语义判别读取的是一个受约束的 decision packet。该 packet 由预定义字段组成，包含五
类信息：
1. 查询视图：原始 query、按子句切分后的 primary request 、supplemental texts 以及引号片
段等局部结构信号。
2. 硬约束规则：primary 必须唯一、primary 与 related 都必须来自候选集、related 不能包含 primary，
以及在不确定时允许升级到协作复核。
3. 软提示信号：混淆来源、召回 top1/top2 线索和 head-score gap 。
4. 候选表：逐行记录候选标识、召回分数、节点类型、层级位置、matched phrases，以及 primary
/ secondary / scene hits 等局部证据。
5. 结构化输出契约：模型需要返回 primary 与 related 选择、 候选级judgement 表、 升级标记，以
及主任务理由、不确定性摘要、混淆点和 challenger 注释等语义交接字段。
这些字段构成后续协作复核的输入契约。尤其是 uncertainty summary 、confusion points 和
challenger notes 一起定义了从直接路由到协作复核的语义交接边界。正文保留融合得分、置信校准
和升级规则；字段清单放在附录，是因为它们属于实现契约而非核心算法 claim。

A.2 角色化复核 Packet
协作复核阶段通过角色专属投影函数 Πr 生成不同视图。 对DomainExpert， 系统保留任务匹配、
正证据卡、top-k 候选和直接路由的主任务摘要；对 GovernanceRisk，系统突出高风险标记、负证
据卡、跨域竞争关系和 override sensitivity ；对 HierarchyResolver，系统强调 parent-child、
base-segment、same-L1 competition 与 matched phrases ；对 UserPreference，系统突出次要诉
求、secondary anchors、related 恢复卡以及 primary-vs-secondary 的歧义说明。角色差异同时体现
在 system prompt 和可见证据子集上。
每个角色的结构化输出都包含两组字段。第一组是候选提案字段：
1. proposal_primary_fqdn
2. proposal_related_fqdns
3. confidence
4. rationale
5. override_position
6. override_basis_tags
第二组是职责维度上的 role signal 五元组：
1. role_signal_kind
2. role_signal_polarity
3. role_signal_target_fqdn
4. role_signal_strength
5. role_signal_note
这些字段共同支持正文中的 round-1/round-2 共识排序和责任式授权检查。
A.3 为何将 Prompt 放入附录
本文的主文只保留决定算法行为的三类对象：候选级分数、升级规则和改判授权规则。完整
prompt 文本、 字段枚举和角色可见视图放在附录， 是因为它们在功能上更接近可复现实验协议， 而
非需要在正文逐句展开的理论核心。对于当前任务，正文若直接插入长 prompt，会稀释方法主线；
若完全省略结构化字段与角色视图，审稿人又无法判断系统是否真的可复现。因此，本稿采用“正文
给公式与控制逻辑，附录给 packet 契约与 prompt 结构”的组织方式。
