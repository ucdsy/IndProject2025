# 专利提炼与技术交底初稿（2026-04-01）

> 用途：从当前 `IndProj04` 项目中提炼 1 项最适合当前阶段提交的发明专利。本文面向内部讨论、技术交底和与代理人沟通，不是最终法律文本。

## 1. 结论先行

### 1.1 推荐主案题目
**一种面向智能体语义寻址与执行发现的受约束多阶段路由决策方法、系统、设备及存储介质**

### 1.2 推荐理由
- 该题目与任务书高度一致，直接覆盖“生成-选择-共识-可信标识”主线。
- 该题目与当前代码实现和评测证据最贴合，不需要硬编尚未落地的机制。
- 该题目能够同时容纳：
  - `routing_fqdn` 与 `agent_fqdn` 两层地址体系；
  - `Stage R / A / B / C` 的多阶段决策链；
  - 候选内共识、职责化 override 和可信轨迹；
  - 最终落到可调用 `endpoint` 的执行闭环。

### 1.3 一句话发明点
不是直接对海量智能体实例做自由检索或开放式多智能体辩论，而是先把自然语言请求约束到稳定的能力地址候选集合，再在冻结候选集合内通过快路径裁决、慢路径异质共识和受控 override 形成最终 `routing_fqdn`，最后再映射为具体 `agent_fqdn` 与 `endpoint`，并全程生成可审计的结构化轨迹。

## 2. 发明点边界

### 2.1 建议写进独立权利要求的核心
- 两层地址分离：
  - `routing_fqdn` 表示能力地址；
  - `agent_fqdn` 表示实例地址；
  - 先路由能力，再发现实例。
- 多阶段受约束路由链：
  - `Stage R` 生成能力地址候选；
  - `Stage A` 进行 primary/related 快路径裁决；
  - `Stage B` 在触发条件下进行候选内异质共识；
  - `Stage C` 基于实例状态、schema、标签和公平因子选择执行者。
- `Stage B` 的关键限制：
  - 只能在冻结候选集合内判断；
  - 不是重新检索，也不能发明新地址；
  - override 必须满足投票、显式证据、敏感场景责任确认等条件。
- 结构化可信轨迹：
  - 记录候选、理由、角色信号、共识得分、override 阻断原因、实例选择依据。

### 2.2 建议作为从属权利要求展开的点
- `Stage A` 的升级触发条件：
  - 低置信度；
  - top1/top2 分差过小；
  - 高风险场景；
  - 主次意图冲突。
- `Stage B` 的角色划分：
  - `DomainExpert`；
  - `GovernanceRisk`；
  - `HierarchyResolver`；
  - `UserPreference`。
- `Stage B` 的角色化视图与语义交接：
  - `primary_rationale`；
  - `secondary_rationale`；
  - `uncertainty_summary`；
  - `challenger_notes`；
  - `confusion_points`。
- `Stage B` 的候选证据卡：
  - 正证据卡；
  - 负证据卡；
  - `secondary_recovery_card`。
- `Stage C` 的排序公式：
  - `base(a)`；
  - `health(a)`；
  - `fair_agent(a)`；
  - `fair_provider(a)`；
  - `final(a)=base*health*fair_agent*fair_provider`。

### 2.3 不建议写进主案中心权利要求的点
- `bandit`、强化学习、在线学习闭环：
  - 当前仓库明确未作为 v1 主线实现。
- 向量库 / HNSW / Embedding 检索：
  - 仓库当前核心贡献不依赖该部分，写入会稀释发明点。
- 完整 A2A / ACP / MCP 协议兼容：
  - 可作为实施环境，不宜当主案核心创新。
- `aggressive replay`：
  - 这是 retrospective 分析结果，不宜写成稳定主技术方案。
- 区块链式 hash 链：
  - 当前文档里更多是可选口径，不宜作为主案必要特征。

## 3. 要解决的技术问题

### 3.1 现有技术主要缺陷
- 现有 Agent 注册/发现方案多聚焦“如何登记”和“如何检索”，难以解决自然语言请求在多能力竞争、主次意图并存、高风险约束触发时的稳定寻址问题。
- 通用多智能体 debate 类方案常允许自由发挥，边界不清，难以保证输出仍落在可验证的候选能力集合内。
- 直接按实例级检索容易把“能力判断”和“实例发现”混在一起，导致：
  - 语义路由不稳定；
  - 评测难统一；
  - 实例状态波动影响能力归因。
- 现有方案普遍缺乏对 override 的责任约束，导致高风险或跨域场景中难以解释“为什么推翻原判”。
- 即使得到候选结果，很多系统也无法输出对审计友好的结构化轨迹。

### 3.2 本项目对应的技术问题表述
如何在智能体服务场景中，将自然语言请求稳定映射到可调用的智能体执行地址，并在主次意图区分、层级竞争、跨域重叠和高风险约束场景下，兼顾路由准确性、可控性、可解释性和实例选择公平性。

## 4. 核心技术方案

### 4.1 总体流程
1. 接收用户查询 `query`、上下文 `context`、约束 `constraints` 与注册快照 `agent_registry_snapshot`。
2. 基于命名空间节点描述进行能力候选召回，得到 `routing_fqdn` 候选集合。
3. 对候选集合执行快路径裁决，输出：
   - `selected_primary_fqdn`；
   - `selected_related_fqdns`；
   - `confidence`；
   - `margin`；
   - `escalation_reasons`。
4. 若满足升级条件，则进入慢路径共识：
   - 构造语义交接信息；
   - 为不同职责角色构造候选视图；
   - 在冻结候选集合内输出各角色提案；
   - 聚合角色信号和反馈得分；
   - 在满足 override 约束时允许推翻快路径，否则维持原判。
5. 得到最终 `routing_fqdn` 后，根据精确 `routing_fqdn` 匹配对应 `agent_fqdn` 集合。
6. 对实例候选执行结构化过滤和排序，输出：
   - `chosen_agent_fqdn`；
   - `endpoint`；
   - `top_k_agents`；
   - `selection_trace`。
7. 生成全链路结构化轨迹，用于解释、复盘和审计。

### 4.2 关键步骤 A：能力地址候选生成
- 召回对象不是 agent card，而是 `namespace descriptor`。
- 命名空间节点描述包含：
  - `fqdn`；
  - `aliases`；
  - `desc`；
  - `industry_tags`；
  - `risk_tags`；
  - `action_tags`；
  - `object_tags`；
  - `parent/children`；
  - `allowed_l3`。
- 候选生成时综合：
  - query 词面别名命中；
  - 描述相似度；
  - context 元数据匹配；
  - 层级重排；
  - 父级回落；
  - 混淆源标记。

### 4.3 关键步骤 B：快路径 primary/related 裁决
- `Stage A` 不是简单 top1 分类，而是同时输出：
  - primary；
  - related；
  - 置信度；
  - primary 与 challenger 的差值；
  - 不确定性摘要。
- `Stage A` 将 query 理解为：
  - `scene_context`；
  - `primary_intent`；
  - `secondary_intents`。
- `Stage A` 还输出交接字段，供 `Stage B` 使用：
  - `primary_rationale`；
  - `secondary_rationale`；
  - `uncertainty_summary`；
  - `confusion_points`；
  - `challenger_notes`；
  - `override_sensitivity`。

### 4.4 关键步骤 C：候选内异质共识
- 仅当快路径不确定或高风险时进入慢路径。
- 慢路径明确限定为“候选内共识”，不允许：
  - 在候选外发明新 `routing_fqdn`；
  - 通过慢路径掩盖 `Stage R miss`。
- 角色集合至少包括：
  - 领域专家；
  - 治理/风险；
  - 层级消歧；
  - 用户偏好。
- 每个角色只看与其职责相匹配的候选视图，避免全员重复看同一信息。
- 每个角色输出：
  - `proposal_primary_fqdn`；
  - `proposal_related_fqdns`；
  - `override_position`；
  - `override_basis_tags`；
  - `role_signal_kind`；
  - `role_signal_polarity`；
  - `role_signal_target_fqdn`；
  - `role_signal_strength`。

### 4.5 关键步骤 D：受控 override
- `Stage B` 不以“谁得分高谁赢”直接结束，而是对 override 增加约束：
  - 票数门限；
  - 与 Stage A 分差门限；
  - 显式支持证据门限；
  - 敏感 override 的二轮复核；
  - 责任角色确认。
- 敏感 override 至少包括：
  - 高风险场景；
  - 层级替换；
  - 跨 L1 领域替换。
- override 被阻断时，系统回退到 `Stage A` 主判，并记录阻断原因。

### 4.6 关键步骤 E：实例发现与公平排序
- 最终执行者不是在前面自由选，而是在最终 `routing_fqdn` 固定后做精确匹配。
- `Stage C` 先过滤：
  - `routing_fqdn` 不完全匹配的实例；
  - 离线或非在线状态实例；
  - 缺失 endpoint 的实例；
  - schema 不兼容的实例。
- 对通过过滤的实例，计算：
  - `base(a)`：能力匹配、schema 匹配、tag 匹配；
  - `health(a)`：在线状态与心跳衰减；
  - `fair_agent(a)`：单实例曝光抑制；
  - `fair_provider(a)`：同 provider 曝光抑制。
- 最终以 `final(a)=base*health*fair_agent*fair_provider` 排序，输出原因拆解。

## 5. 与现有技术的区分建议

### 5.1 相比通用注册/发现方案
- 不是直接按 skill/tag 找 agent，而是先确定稳定能力地址，再决定具体实例。
- 不是单阶段发现，而是“候选生成-快路径-慢路径-实例发现”的分层链路。
- 不是只返回命中列表，而是显式输出 primary 和 related 两类结果。

### 5.2 相比通用多智能体共识方案
- 共识活动被限制在冻结候选集合内，不是开放式生成。
- 角色有职责边界，角色信号会参与 override 放行或阻断。
- 对敏感场景的 override 增加了责任角色确认和二轮复核。
- 系统保留结构化 `trust_trace`，可复盘为何发生或未发生 override。

### 5.3 相比语义检索/向量排序方案
- 本发明不以向量检索本身为核心，而以“受约束路由决策链”作为核心。
- 重点在候选内的主次意图区分、职责化共识和可审计决策，而不只是相关性排序。

## 6. 权利要求书骨架建议

## 6.1 独立权利要求 1：方法
一种面向智能体语义寻址与执行发现的受约束多阶段路由决策方法，其特征在于，包括：
- 获取用户查询、上下文信息和约束信息；
- 基于预设命名空间节点描述，对所述用户查询进行能力地址候选召回，生成至少一个 `routing_fqdn` 候选集合；
- 在所述候选集合内执行快路径裁决，得到主能力地址、相关能力地址、置信度和升级原因；
- 在所述快路径裁决满足预设升级条件时，基于不同职责角色对所述候选集合进行候选内异质共识，得到最终能力地址；
- 基于所述最终能力地址，从智能体注册快照中筛选出与所述最终能力地址精确对应的实例地址集合，并根据实例匹配度、健康度和曝光公平度确定目标实例地址与调用端点；
- 输出所述目标实例地址、调用端点以及对应的结构化决策轨迹。

## 6.2 独立权利要求 2：系统/装置
一种面向智能体语义寻址与执行发现的路由决策系统，包括：
- 候选生成模块；
- 快路径裁决模块；
- 慢路径共识模块；
- override 约束模块；
- 实例发现与排序模块；
- 轨迹记录模块；
其中，各模块协同实现权利要求 1 所述方法。

## 6.3 独立权利要求 3：设备/介质
- 电子设备；
- 计算机可读存储介质；
- 存储有使处理器执行上述方法的指令。

## 6.4 从属权利要求建议
- 根据命名空间节点描述的别名、描述文本、上下文元数据和层级关系生成候选集合。
- 升级条件至少包括置信度阈值、margin 阈值、高风险标识和主次意图冲突标识。
- 慢路径角色至少包括领域专家角色、治理风险角色、层级解析角色和用户偏好角色。
- 每个角色仅接收与职责匹配的候选视图。
- 共识过程中生成正证据卡、负证据卡和 secondary 恢复卡。
- override 需满足票数门限、得分增益门限和显式支持证据门限。
- 在高风险、层级替换或跨领域替换情形下，override 还需满足责任角色确认或二轮复核。
- 实例排序按照实例匹配得分、心跳衰减因子、实例曝光因子和 provider 曝光因子组合生成。
- 所述轨迹至少记录候选集合、角色提案、角色信号、override 阻断原因、实例过滤原因和最终排序依据。

## 8. 可写入说明书的技术效果

> 技术效果可写入说明书“有益效果”与实施例结果，但不应写入权利要求。

### 8.1 可稳妥表述的效果
- 通过将能力路由与实例发现分层，降低实例动态波动对能力判断的干扰。
- 通过快慢路径协同，在普通样本上保持较低慢路径开销，在复杂样本上提供复核能力。
- 通过候选内异质共识和 override 责任约束，提高复杂场景下的可控性与可解释性。
- 通过输出结构化轨迹，支持事后复盘、审计和责任界定。
- 通过实例曝光正则化，降低单实例或单 provider 过度垄断的风险。

### 8.2 当前仓库中可引用的实验数字
- retrospective pooled test（`n=113`）中：
  - `A_llm_v2` 的 `PrimaryAcc@1 = 0.8761`；
  - `A_llm_v2 -> B` 的 `PrimaryAcc@1 = 0.8850`。
- pooled retrospective test 中，`heterogeneous_v3 + aggressive replay` 可达 `0.9292`，但该结果属于 retrospective supplementary，不建议作为主案唯一效果证据。
- canonical frozen result 中：
  - `challenge` 上 `A_llm_v2: 0.6250 -> A_llm_v2 -> B: 0.6667`；
  - `holdout2` 上 `A_llm_v2: 0.8889 -> A_llm_v2 -> B: 0.9074`。

### 8.3 写法建议
- 对外稳妥口径：
  - “在多意图、层级竞争与跨域重叠样本上提升路由稳定性，并提供结构化复核轨迹。”
- 对代理人交底可更具体：
  - “在冻结候选集合条件下，通过职责化慢路径共识实现复杂样本修正，同时保持 changed cases 的零回归。”
