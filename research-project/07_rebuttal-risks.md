# 评审风险清单

## 可能的评审问题
1. 这和已有的 multi-agent debate / voting 有什么本质区别？贡献是否只是工程拼装？
2. 数据与标签是否可信？样本量是否足够？是否存在泄漏（把规则/答案喂给模型）？
3. 你说“可信标识/行为链”，到底可信在哪里？是安全、可审计还是可解释？是否只是日志？
4. 质量提升是否只是因为更多调用/更大成本？单位成本下还成立吗？
5. 方法是否泛化到不同任务/不同模型？还是只对某一业务/某一模型有效？

## 预设回应
- 区分点回答模板:
  - 异质性不是“多次采样”，而是结构化的角色/目标差异 + 可配置机制（给出角色集与生成策略）。
  - 共识不是一次性投票/讨论，而是“反馈驱动的演化”，用用户代理/标签反馈函数对候选决策打分，形成可分析的收敛过程（给出算法伪代码 + 消融）。
  - 可信标识限定为“可审计/可核验的过程完整性”：定义日志 schema、字段完整率、hash-chain 校验与篡改检测演示；不夸大为强安全。
- 数据可信性:
  - 强调标注流程；公布标签体系/命名空间说明与标注指南；提供 train/test 分割策略。
  - 若样本量小，补人评/专家评审并展示一致性（inter-annotator agreement）。
- 成本公平:
  - 报告 token/latency；给出“质量-成本曲线”；在固定预算下比较不同策略（同 token 上限）。
- 泛化性:
  - 做跨任务小实验（哪怕 50-100 条）或跨模型复现实验（同提示/同框架）。

## 尚缺实验
- 固定 token budget 下的对比（避免“堆调用”质疑）。
- 小规模泛化验证（路由主任务主表 + 公开工具路由/意图到工具选择 mini 表）。
- 人评协议与一致性统计（解释质量、可控性、审计可用性）。
